使用 Obsidian 搭配 hexo 与阿里云 oss 搭建博客系统

前段时间看了刘未鹏的《暗时间》这本书后决心开始写博客，虽然是设计师出身，但骨子里还是对技术有狂热追求。

原本是想直接在少数派的 Mtrix 社区直接写了发表，但是有一些硬核或是自认为没有什么价值的文章可能会影响我在少数派写作的数据，所以还是决定自己搭建一个静态博客，文章放在本地，想发布的时候就发布，使用一些自动化工具，即便没有数据库，也能通过目录来管理博客内容。

搭配 Obsidian 使用 Hexo#

已经习惯了在 Obsidian 中写东西，也希望把 OBS 作为 All in one 的入口，所以无论是链接到别处的多维表格，还是链接在本地目录的某个设计源文件，都先通过一个 md 文档来建立起基本的项目信息，要搭建博客，也会优先考虑纯静态配合 md 文档的 Hexo。

我的 Obsidian 使用了 iCloud 同步，并不适合使用 git 发布，因此我在笔记库中新建一个文件夹，专门用来发布文章到 Hexo，这过程中就需要创建一个软链接，将 Hexo 项目目录中的 source/_posts 指向该文件夹，在使用 Hexo 生成目录的时候，就会自动检索被其链接的 Obsidian 文件夹下的 md 文档了。

但是使用软连接的方法并没有生效，最终使用 ln -s 创建硬链接解决。 完整的指令如下：

ln -s "~/Library/Mobile Documents/iCloud~md~obsidian/Documents/Ivan’s wiki/100-Project/博客文章" "~/Desktop/Hexo/source/_post"

此时再使用 npx hexo g 生成即可准确识别到 md 文档，下一步是自动发布到阿里云的 OSS。

部署到 OSS#

因为阿里云 OSS 的首页是简历，所以博客部分通过创建一个新的 blog 子文件夹来存放，而发布过程需要一个 hexo 部署插件：hexo-deployer-ali-oss

  • 执行模块的安装：npm i hexo-deployer-ali-oss --save
  • 修改配置文件 _config.yml 的 deploy 部分内容
deploy:
  type: 'ali-oss'
  region: 'oss-cn-beijing'
  accessKeyId: 'ak'
  accessKeySecret: 'sk'
  bucket: 'bucket名称'
  remotePath: '子目录'

之后执行 npx hexo d 即可自动部署到 OSS 的 blog 目录下，而将 hexo g & hexo d 加入到 package.json 中的 script 中，就可以实现一条命令自动生成文章 + 部署到 OSS。

遇到了一些问题#

  1. 整个博客首页打开后丢失了样式，内容错位
  2. Mac 系统下的 .DS _Store 隐藏文件也会被上传

很显然第一个问题是使用了绝对路径的事，而第二个问题就需要去原项目看看有没有相关参数可以让其忽略某些特定文件。

在 Hexo 的配置文件中，把域名 https://ivan.show/blog 填入后，样式的问题得到解决。

而第二个问题找到作者的项目后，发现是一个四年前的项目，代码量很少，翻看一下决定自己改一下代码处理，找到 hexo 项目中的 node_modules/hexo-deployer-ali-oss/index.js 这个文件。有这么一段是遍历目录：

    rd.eachFileSync(localDir, function (f) {
        // 每找到一个文件都会调用一次此函数
        put(f)
    });

这么看，大概有两个方法可以解决，一个是在执行 put 函数前判断文件名是否为 .DS _Store ，或是在 put 函数内部入口处进行判断。这里就在遍历时做一个判断，直接问一下 GPT：

执行 npm run blog 试一下，发现已经可以跳过 .DS _Store 文件了，此时还需进入 OSS 的 blog 目录，删除掉原来的全部文件，再上传部署一次就可以了。

使用 Obsidian 模板新建文章#

在 Obsidian 中模板目录下，创建一个 `hexo 模板 ，并写下以下内容：

---
title: 示例标题
date: {{date}}
---

如此一来，在创建一篇新的文章时，使用 插入模板 命令选择 hexo模板 即可一键插入头部信息。

使用 Alfred 一键执行部署#

在 Obsidian 中写完的文章，想必不希望再到终端或是 VSCode 项目中去执行 npm 命令。我习惯了使用 Alfred 操作众多快捷的指令，做了一个简单的 workflow，通过输入 blog 一键更新部署。

至此，整个从写作到发布的工作流就算梳理完成，接下来就是重新设计 UI，优化主题了。
